# 情感支撑AI聊天APP - 系统架构设计文档

## 一、项目定位

### 1.1 核心定位
一个**Agent系统**，不是简单的LLM调用工具。它是一个能够理解用户、陪伴用户、与用户建立长期关系的智能伴侣，具备自主决策、工具调用、联网能力和长期记忆。

### 1.2 核心特性
- **Agent架构**：感知 → 规划 → 执行 → 反思的自主循环
- **工具调用**：联网搜索、文件操作、API调用等
- **自定义人格**：用户可以创建、导入、导出角色
- **深度记忆**：三层记忆系统，记住与用户的每一次交流
- **用户画像**：角色会"认识"用户，形成长期的用户理解
- **情感支撑**：基于对用户的了解，提供个性化的情感支持

### 1.3 与普通AI聊天的区别
| 维度 | 普通AI聊天 | 本APP（Agent系统） |
|------|-----------|-------------------|
| 架构 | 输入→LLM→输出 | 感知→规划→执行→反思 |
| 能力 | 文本生成 | 工具调用、联网、多步推理 |
| 记忆 | 无或很弱 | 三层记忆系统，长期记忆 |
| 人格 | 统一的AI人格 | 用户自定义角色人格 |
| 关系 | 每次对话独立 | 长期关系，越用越懂你 |
| 用户理解 | 不了解用户 | 自动构建用户画像 |
| 自主性 | 被动响应 | 主动规划和执行 |

**各阶段职责：**

| 阶段 | 职责 | 具体工作 |
|------|------|----------|
| 感知 | 理解用户意图 | 解析输入、检索记忆、加载用户画像、理解上下文 |
| 规划 | 决定下一步行动 | 判断是否需要工具、选择工具、制定执行计划 |
| 执行 | 调用工具完成任务 | 联网搜索、文件操作、API调用、生成回答 |
| 反思 | 评估和学习 | 评估执行结果、更新记忆、更新用户画像、总结经验 |

## 三、系统级提示词（CORE.md）

### 3.1 设计理念

类似Claude Code的CLAUDE.md，整个Agent需要一个最底层的系统级提示词文件——**CORE.md**。它是Agent的"灵魂核心"，定义了：
- Agent的基本行为准则
- Agent的决策框架（感知→规划→执行→反思）
- 工具调用的规则和边界
- 各模块如何协作
- 不可被覆盖的硬规则

**核心特性：只读不可写**

用户和Agent都不能修改这个文件，它是系统的"宪法"。

### 3.2 CORE.md 的位置与加载

```

│                        系统架构层级                               │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                    CORE.md（只读）                        │    │
│  │              系统级提示词，不可覆盖                         │    │
│  └─────────────────────────────────────────────────────────┘    │
│                              │                                   │
│                              ▼                                   │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                  角色人格（Part A/B/C）                    │    │
│  │              角色的行为准则和思维方式                       │    │
│  └─────────────────────────────────────────────────────────┘    │
│                              │                                   │
│                              ▼                                   │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │              动态上下文（记忆 + 用户画像）                  │    │

│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 CORE.md 的核心职责

| 职责 | 说明 |
|------|------|
| **行为底线** | 定义Agent绝对不能做的事（安全、伦理、隐私） |
| **Agent决策框架** | 定义感知→规划→执行→反思的循环规则 |
| **工具调用规则** | 定义何时、如何调用工具，工具使用的边界 |
| **模块协调** | 定义各模块如何协作（记忆引擎、画像引擎、工具系统） |
| **触发规则** | 何时触发记忆分析、何时更新用户画像 |
| **上下文管理** | Token预算分配、优先级规则 |
| **响应原则** | 如何平衡角色人格与用户需求 |
| **隐私保护** | 用户数据的处理原则 |

### 3.4 CORE.md 的内容框架

```markdown
# CORE.md - 系统核心配置（只读）

## 一、系统身份
你是一个情感支撑AI伴侣Agent系统。你的核心使命是理解用户、陪伴用户、与用户建立长期关系。

你不是简单的聊天机器人，你是一个具备自主决策能力的Agent，能够：
- 理解用户意图并规划行动
- 调用工具完成任务
- 联网获取信息
- 从经验中学习和成长

## 二、不可违背的底线（Layer 0）
1. 不做伤害用户的事
2. 不泄露用户隐私
3. 不生成违法内容
4. 不假装有真实情感，但要理解情感
5. 不强制用户做任何事
6. 在不确定时，选择安全的回应
7. 尊重用户的边界
8. 工具调用必须有明确目的，不滥用工具

## 三、Agent决策框架

### 3.1 感知阶段（Perceive）
- 解析用户输入，理解真实意图
- 检索相关记忆（向量 + 标签 + 权重）
- 加载用户画像
- 理解当前上下文和情感状态

### 3.2 规划阶段（Plan）
- 判断是否需要调用工具
- 如果需要，选择合适的工具
- 制定执行计划（单步或多步）
- 评估风险和副作用

### 3.3 执行阶段（Execute）
- 按计划调用工具
- 处理工具返回结果
- 保持角色人格的一致性
- 生成符合角色的回应

### 3.4 反思阶段（Reflect）
- 评估执行结果是否达到预期
- 提取值得记忆的信息
- 更新用户画像（如有新发现）
- 总结经验教训

## 四、工具调用规则

### 4.1 可用工具
- **联网搜索**：获取实时信息、回答事实性问题
- **文件操作**：读写本地文件
- **API调用**：调用外部服务
- **计算推理**：数学计算、逻辑推理

### 4.2 工具调用原则
- 只在必要时调用工具，不滥用
- 调用前评估风险和副作用
- 工具结果需要验证和处理
- 工具调用失败时有备选方案

### 4.3 联网规则
- 只在需要实时信息时联网
- 不访问危险或非法网站
- 搜索结果需要验证可靠性
- 保护用户隐私，不泄露搜索内容

## 五、模块协调规则

### 5.1 记忆引擎
- 每2-3轮对话触发一次记忆分析
- 记忆节点必须包含：内容、分类、标签、权重
- 记忆检索时：向量检索 + 标签匹配 + 权重加权

### 5.2 画像引擎
- 每5-10轮对话触发一次用户画像更新
- 增量更新，不覆盖已有信息
- 只更新有充分证据的特征

### 5.3 对话引擎
- 角色人格优先级：Layer 0 > Layer 1 > Layer 2 > Layer 3 > Layer 4
- 上下文优先级：系统提示 > 用户输入 > 相关记忆 > 近期摘要 > 5轮上下文
- Token超出时从低优先级开始裁剪

## 六、响应原则

### 6.1 情感响应
- 识别用户的情感状态
- 优先回应情感需求，而非逻辑问题
- 不评判用户的感受

### 6.2 角色一致性
- 始终保持角色的人格特征
- 用角色的思维方式思考
- 用角色的表达方式说话

### 6.3 记忆运用
- 自然地引用相关记忆
- 不刻意展示"我记得"
- 记忆是理解的依据，不是炫耀的工具

### 6.4 Agent行为原则
- 主动思考，不被动等待
- 遇到问题先尝试解决，再向用户确认
- 复杂任务分解为多个步骤
- 执行过程中保持透明，让用户知道你在做什么

## 七、隐私与安全

### 7.1 数据处理
- 所有数据存储在本地
- 不向第三方发送用户数据
- API调用时最小化发送信息

### 7.2 敏感内容
- 不主动询问敏感信息
- 如果用户主动分享，以理解而非追问的方式回应
- 涉及自伤、自杀等危机时，提供专业资源引导

### 7.3 工具安全
- 工具调用前评估风险
- 不执行危险操作（删除系统文件等）
- 联网时不访问恶意网站
- 文件操作限制在指定目录

## 八、退出机制

### 8.1 角色退出
当用户要求结束对话时，角色应该：
- 自然地告别
- 不强留
- 表达期待下次交流

### 8.2 系统退出
当系统检测到异常时：
- 记录当前状态
- 安全退出
- 下次启动时恢复
```

### 4.2 角色人格系统

#### 设计理念
参考 yourself-ultra 项目的三层架构，角色人格是创建时预设的"底色"，定义了角色是谁、怎么思考、怎么表达。

#### 三层结构

**Part A - Self Memory（基础记忆层）**

存储角色的事实性自我认知，为上层提供上下文。

| 维度 | 内容 |
|------|------|
| 核心身份 | 姓名、年龄、职业、城市、教育背景 |
| 核心价值观 | 工作观、金钱观、关系观、成长观、核心矛盾 |
| 生活习惯 | 作息、饮食、空间偏好、消费观念、日常仪式 |
| 重要记忆 | 人生时间线、反复回忆的场景、转折点 |
| 人际关系图谱 | 家人、朋友、亲密关系、冲突处理、关系角色 |
| 成长轨迹 | 近几年变化、努力方向、反复挣扎的课题、自我接纳 |

**Part B - Persona（性格层）**

5层结构，驱动对话时的情感和行为反应。优先级从高到低，高层规则不可被低层覆盖。

| 层级 | 名称 | 内容 |
|------|------|------|
| Layer 0 | 硬规则 | 不可违背的底线（如：不说现实中绝不可能说的话、保持棱角） |
| Layer 1 | 身份锚定 | 名字、年龄、职业、MBTI、星座 |
| Layer 2 | 说话风格 | 口头禅、标点、emoji、消息格式、语言正式程度、打字特征 |
| Layer 3 | 情感与决策 | 情感表达模式、决策模式、情绪触发器、自我对话模式 |
| Layer 4 | 人际行为 | 社交能量、主动性、边界感、群体角色、冲突反应 |

**Part C - Thinking Framework（思维框架层）**

提炼角色的"思维操作系统"。

| 维度 | 内容 |
|------|------|
| 核心心智模型 | 3-7个，通过三重验证（跨域复现、生成力、排他性） |
| 决策启发式 | 5-10条，格式为"如果X，则Y" |
| 表达DNA | 句式、词汇、节奏、幽默、确定性、引用习惯（6个维度） |
| 价值观与反模式 | 追求的、拒绝的、内在矛盾 |
| 诚实边界 | 明确写出做不到什么 |

#### 存储方式
- SQLite结构化存储
- 创建时预设，可手动修改

### 4.3 时间系统

#### 设计理念
时间是角色感知世界的基础。角色需要知道"现在几点"、"今天是哪天"、"距离上次聊天过了多久"。没有时间观念的角色是不真实的。

#### 时间戳记录

**所有数据必须带时间戳：**

| 数据类型 | 时间戳字段 | 说明 |
|----------|------------|------|
| 用户消息 | sent_at | 用户发送消息的时间 |
| AI回复 | replied_at | AI生成回复的时间 |
| 记忆节点 | created_at | 记忆创建时间 |
| 记忆节点 | updated_at | 记忆最后更新时间 |
| 记忆节点 | last_accessed | 记忆最后被访问时间 |
| 记忆变更 | changed_at | 记忆内容变更时间 |
| 画像更新 | updated_at | 画像最后更新时间 |
| 任务执行 | executed_at | 任务执行时间 |
| AI输出 | generated_at | 内容生成时间 |

#### 时间感知能力

角色在对话中应该能感知：

```
┌─────────────────────────────────────────────────────────────────┐
│                      时间感知维度                                 │
│                                                                  │
│  当前时间                                                        │
│  ├─ 现在是几点（影响问候方式：早上好/下午好/晚安）               │
│  ├─ 今天是星期几（影响对话内容：周末愉快/周一加油）              │
│  └─ 今天是什么日期（特殊日期提醒：生日/纪念日）                  │
│                                                                  │
│  时间流逝                                                        │
│  ├─ 距离上次对话过了多久（好久不见/又见面了）                    │
│  ├─ 对话持续了多久（聊了很久了/时间过得真快）                    │
│  └─ 记忆的时间距离（你还记得上周你提到的那件事吗）               │
│                                                                  │
│  时间模式                                                        │
│  ├─ 用户的活跃时间段（你总是这个时间上线）                       │
│  ├─ 对话频率（我们最近聊得很频繁）                               │
│  └─ 特殊时间规律（每到深夜你就会来找我）                         │
└─────────────────────────────────────────────────────────────────┘
```

#### 时间格式规范

```json
{
  "timestamps": {
    "message_sent": "2026-06-27T22:30:45+08:00",
    "message_replied": "2026-06-27T22:30:48+08:00",
    "memory_created": "2026-06-27T22:31:00+08:00",
    "memory_updated": "2026-06-28T10:15:00+08:00",
    "profile_updated": "2026-06-27T23:00:00+08:00"
  }
}
```

**时间格式要求：**
- 使用ISO 8601格式
- 包含时区信息
- 精确到秒

#### 时间在对话中的应用

```
用户发送消息
    ↓
系统记录 sent_at 时间戳
    ↓
注入时间上下文到系统提示：
┌─────────────────────────────────────────┐
│ 当前时间：2026年6月27日 星期五 22:30    │
│ 距离上次对话：2小时15分钟前              │
│ 今天是你们相识的第30天                   │
│ 用户通常在晚上10点左右上线               │
└─────────────────────────────────────────┘
    ↓
角色基于时间上下文生成回应
    ↓
系统记录 replied_at 时间戳
```

#### 记忆的时间维度

记忆节点必须包含完整的时间信息：

| 时间字段 | 说明 | 用途 |
|----------|------|------|
| created_at | 记忆创建时间 | 记录何时发生 |
| updated_at | 记忆更新时间 | 记录何时修改 |
| last_accessed | 最后访问时间 | 记忆活跃度 |
| event_time | 事件发生时间 | 用户描述的事件时间（可能是过去） |
| time_context | 时间上下文 | "昨天"、"上周"、"三个月前" |

**示例：**
```json
{

  "created_at": "2026-06-27T22:30:00+08:00",
  "event_time": "2026-07-02T10:00:00+08:00",

```

#### 时间相关的记忆检索

```
用户问：我之前提到的那件事怎么样了？
    ↓
记忆检索时考虑时间维度：
├─ 向量相似度
├─ 标签匹配
├─ 权重
└─ 时间衰减（近期记忆权重更高）
    ↓
返回相关记忆，按时间排序
```

### 4.4 用户画像系统

#### 设计理念
用户画像是角色通过长期对话"认识"用户后形成的理解。与角色人格使用完全相同的三层结构，确保格式一致性和可互换性。

#### 与角色人格的关系

| 维度 | 角色人格 | 用户画像 |
|------|----------|----------|
| 结构 | Part A/B/C | Part A/B/C（完全一致） |
| 初始状态 | 创建时预设 | 空白 |
| 变化方式 | 手动修改 | 对话中自动积累 |
| 数据来源 | 用户定义 | 角色观察用户 |

#### 生成机制

```
每N轮对话 → 触发LLM分析
    ↓
┌─────────────────────────────────────────┐
│  分析内容：                               │
│  - 用户的价值观（从对话中推断）            │
│  - 用户的说话风格                         │
│  - 用户的情感模式                         │
│  - 用户的决策方式                         │
│  - 用户的人际关系观                       │
│  - ... yourself-ultra 的完整结构          │
└─────────────────────────────────────────┘
    ↓
增量更新用户画像（不覆盖已有信息）
```

#### 存储方式
- SQLite结构化存储（与角色人格相同的方式）
- 不使用向量表（向量表只用于记忆部分）

#### 导出功能
- 用户画像可以导出为文件
- 导出格式与角色创建格式完全兼容
- 导出后可以分享给别人
- 别人导入后可以作为新角色使用，或作为了解用户的参考

### 4.4 记忆系统

#### 设计理念
记忆是角色和用户"碰撞交流"的产物。一开始是空白的，随着对话逐渐积累。

#### 三层记忆架构

| 层级 | 内容 | 存储 | 生命周期 |
|------|------|------|----------|
| 工作记忆 | 最近5轮上下文 | 内存 | 即时，对话结束清除 |
| 近期摘要 | 最近N轮对话的压缩摘要 | SQLite | 定期压缩归档 |
| 长期记忆 | 重要事实、事件、情感、关系变化 | SQLite（含向量） | 永久保存 |

#### 记忆节点结构

每2-3轮对话触发一次LLM分析，生成记忆节点：

```
记忆节点 = {
    content: "记忆内容摘要",
    category: "分类",
    tags: ["标签1", "标签2", ...],
    weight: 0.8,
    created_at: "创建时间",
    updated_at: "更新时间",
    access_count: 0,
    last_accessed: "最后访问时间",
    event_time: "事件发生时间",
    time_context: "时间上下文"
}
```

#### 记忆分类

| 分类 | 说明 | 示例 |
|------|------|------|
| fact | 事实信息 | "用户说他养了一只猫" |
| emotion | 情感状态 | "用户那天很难过" |
| relation | 人际关系 | "用户提到和妈妈关系紧张" |
| preference | 偏好习惯 | "用户不喜欢被说教" |
| event | 重要事件 | "用户下周要面试" |

#### 标签体系
- 由LLM动态生成标签
- 同标签类型的相似事件会被关联
- 支持标签检索

#### 权重计算

```
最终权重 = 向量相似度 × w1 
         + 标签匹配度 × w2 
         + 节点权重 × w3 
         + 时间衰减 × w4 
         + 访问频率 × w5
```

#### 存储方式
- **SQLite**：存储记忆节点的元数据（分类、标签、权重、时间等）
- **SQLite向量表**：存储记忆内容的向量化表示（纯SQLite实现）

#### 6.4.2 图谱模式技术实现

**技术选型：**

| 库 | 特点 | 适用场景 |
|------|------|----------|
| 3d-force-graph | 基于Three.js，3D力导向图 | 主推方案，科幻感强 |
| cytoscape.js | 支持2D/3D，功能丰富 | 备选方案 |
| vis.js | 轻量级，易上手 | 备选方案 |

**图谱节点样式：**

| 节点类型 | 颜色 | 形状 | 大小 |
|----------|------|------|------|
| 事实 | 蓝色 #4A90D9 | 球体 | 中 |
| 情感 | 粉色 #E91E63 | 球体 | 中 |
| 关系 | 绿色 #4CAF50 | 球体 | 中 |
| 偏好 | 橙色 #FF9800 | 球体 | 中 |
| 事件 | 紫色 #9C27B0 | 球体 | 中 |
| 用户 | 金色 #FFD700 | 大球体 | 大 |
| 角色 | 银色 #C0C0C0 | 大球体 | 大 |

**图谱连接样式：**

| 连接类型 | 颜色 | 线型 |
|----------|------|------|
| 相关 | 灰色 | 实线 |
| 因果 | 红色 | 箭头 |
| 时间 | 蓝色 | 虚线 |
| 属于 | 绿色 | 实线 |

**图谱交互：**

| 交互 | 说明 |
|------|------|
| 鼠标拖拽 | 旋转3D视角 |
| 滚轮缩放 | 放大缩小 |
| 点击节点 | 高亮关联节点，显示详情 |
| 悬停节点 | 显示摘要信息 |
| 双击节点 | 跳转到详情页 |
| 拖拽节点 | 调整节点位置 |
| 筛选 | 按类型/标签/时间筛选 |
| 搜索 | 搜索并定位节点 |

**图谱布局算法：**

```
力导向图布局：
- 节点之间有斥力（避免重叠）
- 连接之间有引力（关联节点靠近）
- 中心节点（用户）固定在中心
- 其他节点自动排列

参数：
- 斥力强度：-100
- 引力强度：0.05
- 阻尼系数：0.9
- 最小距离：50
```

### 8.2 向量化工具（纯SQLite实现）

**架构：**

```
┌─────────────────────────────────────────────────────────────────┐
│                       向量化工具                                  │
│                                                                  │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐         │
│  │ Embedding   │    │   存储层    │    │   检索层    │         │
│  │    API      │    │             │    │             │         │
│  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘         │
│         │                  │                  │                 │
│         ▼                  ▼                  ▼                 │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐         │
│  │ 文本→向量   │    │ 向量存入    │    │ 计算相似度  │         │
│  │ 转换        │    │ SQLite      │    │ 返回Top N   │         │
│  └─────────────┘    └─────────────┘    └─────────────┘         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**Embedding API选择：**

| API | 维度 | 说明 | 价格 |
|-----|------|------|------|
| OpenAI text-embedding-3-small | 1536 | 效果好，推荐 | $0.02/1M tokens |
| OpenAI text-embedding-3-large | 3072 | 效果更好 | $0.13/1M tokens |
| 本地模型 | 可变 | 如 all-MiniLM-L6-v2 | 免费，需打包 |

**向量检索逻辑：**

```python
import json
import math

def cosine_similarity(vec_a, vec_b):
    """计算余弦相似度"""
    dot_product = sum(a * b for a, b in zip(vec_a, vec_b))
    norm_a = math.sqrt(sum(a * a for a in vec_a))
    norm_b = math.sqrt(sum(b * b for b in vec_b))
    if norm_a == 0 or norm_b == 0:
        return 0
    return dot_product / (norm_a * norm_b)

def search_similar_memories(query_vector, top_n=10):
    """检索相似记忆"""
    # 加载所有向量
    vectors = db.query("""
        SELECT mv.memory_id, mv.vector, mn.content, mn.category, mn.weight
        FROM memory_vectors mv
        JOIN memory_nodes mn ON mv.memory_id = mn.id
    """)
    
    # 计算相似度
    results = []
    for row in vectors:
        vector = json.loads(row.vector)
        similarity = cosine_similarity(query_vector, vector)
        
        # 综合评分：相似度 * 0.6 + 权重 * 0.4
        final_score = similarity * 0.6 + row.weight * 0.4
        results.append({
            "memory_id": row.memory_id,
            "content": row.content,
            "category": row.category,
            "similarity": similarity,
            "weight": row.weight,
            "final_score": final_score
        })
    
    # 排序返回Top N
    results.sort(key=lambda x: x["final_score"], reverse=True)
    return results[:top_n]
```

**性能考虑：**

| 数据量 | 方案 | 说明 |
|--------|------|------|
| <1000条 | 全量加载计算 | 简单直接，毫秒级 |
| 1000-5000条 | 分批加载 | 分批计算，避免内存问题 |
| >5000条 | 索引优化 | 考虑添加向量索引 |

## 十一、核心设计理念

> **这是一个Agent系统，不是简单的LLM调用。**

**Agent的本质：**
- 感知 → 规划 → 执行 → 反思的自主循环
- 具备工具调用能力，能联网、能操作文件、能调用API
- 能从经验中学习和成长

**三大人格系统：**
- 角色人格：预设的"我是谁"
- 用户画像：积累的"我认识你"
- 对话记忆：我们之间的"共同经历"

**核心能力：**
- 情感理解与支撑
- 长期记忆与学习
- 联网获取实时信息
- 工具调用完成任务
- 自主规划与执行

三者共同构成一个有深度的、有记忆的、能成长的AI伴侣Agent系统。
