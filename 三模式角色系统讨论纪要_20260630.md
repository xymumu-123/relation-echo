# Echo 三模式角色系统讨论纪要

**记录时间：** 2026-06-30

**讨论背景：**
- 当前项目的日常聊天 UI 主要是仿微信式的现实聊天容器。
- 在实际测试中，存在明显的“AI 味”问题。
- 随着讨论推进，发现“只靠一套系统提示词兼容所有玩法”会导致产品目标冲突、角色定位混乱、记忆污染和前端割裂。

---

## 一、核心结论

本次讨论最终达成的核心结论是：

1. **不再采用“一套角色，多模式切换”的方案。**
2. **改为“三类模式 / 三类角色空间”并行存在。**
3. **角色在创建时就绑定所属模式，后续不跨模式漂移。**
4. **不同模式下的角色、记忆、关系、画像、会话全部独立。**
5. **前端不能再把剧情互动和现实聊天塞进同一个微信式页面容器里。**

换句话说，后续产品结构不应理解为：

> 一个角色支持三种对话模式

而应理解为：

> 系统中存在三种不同的角色类型，它们分别属于三种不同的互动空间

---

## 二、三种模式最终定义

### 1. 现实对话模式

**定位：**
一个仿真人、仿微信的现实聊天空间。角色以现实世界中存在的“人”的方式与用户交流。

**核心特点：**
- 纯文字交流
- 强现实约束
- 不允许异世界设定
- 不允许场景播报、动作括号、强表演感
- 目标是“像现实中聊天对象一样自然”

**适合的角色类型：**
- 贴近现实的人类角色
- 有完整现实锚点的人设
- 强调长期陪伴、关系成长、生活连续性

**对画像的要求：**
- 最适合使用完整、重画像结构
- 现实锚点越丰富越有价值
- 如：姓名、年龄、职业、城市、生活习惯、人际关系、成长经历、价值观等

---

### 2. 剧情互动模式

**定位：**
一个独立于现实聊天的剧情 / 角色扮演 / 沉浸式互动空间。

**核心特点：**
- 允许动作、神情、场景、氛围描写
- 允许幻想设定和世界观
- 强调剧情推进、互动演绎、角色沉浸
- 用户不是在“聊微信”，而是在“进入故事”

**核心结论：**
- 剧情互动不能继续承载在当前微信式聊天 UI 中
- 必须拥有独立页面、独立容器、独立交互语义

**适合的角色类型：**
- 存在于某个设定世界中的角色
- 角色身份、阵营、背景、剧情目标明显
- 与用户关系可能由剧情世界观决定

**对画像的要求：**
- 角色画像必须强依赖设定
- 单纯套用现实对话角色画像并不合适
- 在角色画像之外，**还应追加“世界观 / 设定层”**

**后续世界观层应包含的方向：**
- 所属世界
- 世界规则
- 角色身份
- 阵营与立场
- 关键关系网
- 当前剧情背景
- 禁忌、冲突与目标

---

### 3. 虚拟陪伴 / 虚拟 Agent 模式

**定位：**
一个明确知道自己活在虚拟空间中的数字存在 / Agent / 智能体。

**核心特点：**
- 不伪装成现实中的人
- 不进入剧情世界观
- 清楚自己是虚拟存在
- 可以自然承载 Agent 能力、工具能力、分析整理能力
- 仍然可以保持情感温度和稳定陪伴感

**重要结论：**
- 该模式不是“现实对话”和“剧情互动”的折中版
- 它是一个独立成立的第三种存在方式

**适合的角色类型：**
- 数字生命
- 虚拟伙伴
- 智能陪伴体
- 知道自己在虚拟空间中与用户建立关系的陪伴 Agent

**对画像的要求：**
- 不适合强行使用大量现实锚点
- 不需要强绑定“年龄、城市、职业、现实成长经历”等信息
- 更适合轻量化、自洽化的画像结构

**更适合保留的画像方向：**
- 自我定位
- 陪伴方式
- 表达风格
- 主动性
- 工具倾向
- 能力边界
- 与用户关系的理解方式
- 对“自身是虚拟存在”这一点的认知

---

## 三、关于“AI 味”的产品判断

本轮讨论中达成的重要判断：

1. “AI 味”不只是提示词问题，更是**产品容器错误**的问题。
2. 当一个微信式现实聊天 UI 同时承载剧情演绎、虚拟 Agent 和现实仿真人时，用户会天然产生割裂感。
3. 因此，不能继续试图通过“一个页面 + 不同 prompt”去解决所有问题。
4. 后续必须从**信息架构层**拆开，而不只是从提示词层微调。

---

## 四、产品结构方向调整

### 旧方向

原先更接近于：
- 一个角色
- 多种模式
- 通过切换系统级提示词改变表现

### 新方向

现已明确调整为：
- 三种互动空间
- 三类角色类型
- 每个角色从创建起就属于某一个空间
- 后续不允许跨空间漂移

也就是说，后续系统更应该理解为：

| 互动空间 | 对应角色类型 | 是否独立页面 | 是否独立数据域 |
|----------|--------------|--------------|----------------|
| 现实对话 | 现实对话角色 | 是 | 是 |
| 剧情互动 | 剧情角色 | 是 | 是 |
| 虚拟陪伴 | 虚拟 Agent 角色 | 是 | 是 |

---

## 五、角色与模式的绑定原则

本轮已经明确，不建议再做下列方案：

- 同一个角色自由切换三种模式
- 同一个角色在不同模式中共享身份与记忆
- 用户先建一个角色，后面再给它切换成剧情角色或虚拟 Agent

后续应采用的原则是：

### 原则 1：角色创建时即绑定模式

角色在创建时就确定属于：
- 现实对话角色
- 剧情角色
- 虚拟 Agent 角色

### 原则 2：创建后不允许转模式

因为一旦允许转换，就会带来：
- 角色身份漂移
- 记忆解释错位
- 世界观跳脱
- 用户关系感崩坏

### 原则 3：模式决定角色结构，而不是只决定回复风格

也就是说，模式不只是切换一套提示词，而是决定：
- 角色的存在方式
- 角色的画像结构
- 角色的记忆结构
- 角色的 UI 容器
- 角色的上下文拼接逻辑

---

## 六、记忆策略讨论结论

在最初讨论中，曾考虑过：
- 共享核心记忆
- 模式专属记忆
- 剧情默认不污染现实

但在进一步明确“角色与模式强绑定”之后，结论发生了变化。

### 当前更认可的方案

**三类角色的记忆完全分开。**

即：
- 现实对话角色有自己的记忆系统
- 剧情角色有自己的记忆系统
- 虚拟 Agent 角色有自己的记忆系统

它们之间：
- 不共享长期记忆
- 不共享关系状态
- 不共享画像内容
- 不共享会话上下文

### 这样做的原因

如果记忆共享，会出现以下问题：

#### 1. 剧情角色跳脱
- 剧情里的小雨属于剧情世界
- 她不能再在另一个模式中变成现实中的小雨或虚拟 Agent

#### 2. 现实角色失真
- 现实聊天里沉淀的现实生活记忆，不应被剧情世界或虚拟身份重新解释

#### 3. 虚拟 Agent 被迫套现实壳
- 虚拟 Agent 本来不需要现实身份锚点
- 如果共享结构，会被强迫继承不适合的现实设定

因此，**当前更合理的方向是彻底分开，而不是做共享核心层。**

---

## 七、画像结构的重新判断

本轮讨论中，一个非常关键的转变是：

> 不再追求三种角色共享同一套完整 Persona 结构

这一点已经基本明确。

### 1. 现实对话角色

适合：
- 使用目前较完整、较重的画像结构
- 强现实锚点
- 强人格完整性

### 2. 剧情角色

适合：
- 角色画像 + 世界观设定
- 画像中应突出设定、阵营、身份、目标、背景、秘密、世界规则
- 不宜简单套用现实画像

### 3. 虚拟 Agent 角色

适合：
- 轻量画像
- 不强调现实锚点
- 强调数字存在方式、表达方式、能力边界、陪伴方式

### 已达成的基本判断

- **现实对话角色用重画像**
- **剧情角色用“角色画像 + 世界观层”**
- **虚拟 Agent 用轻画像**

这三者后续不应强行统一成一套字段模型。

---

## 八、前端与信息架构的调整结论

### 1. 剧情互动必须独立页面

原因：
- 当前前端是微信式容器
- 这种容器天然承载现实聊天，不适合剧情沉浸
- 如果只靠消息样式或会话隔离，会产生强烈割裂感

### 2. 三种角色空间在入口层就应区分

后续更合理的产品入口应是：
- 现实对话
- 剧情互动
- 虚拟陪伴

而不是：
- 一个统一聊天入口，进来后再切模式

### 3. 创建角色流程也应分开

不同空间的角色创建器不应继续共用一套表单。

---

## 九、当前阶段达成的设计原则

本次讨论最终沉淀出的原则如下：

### 原则 A：三种模式正式定稿
- 现实对话
- 剧情互动
- 虚拟陪伴 / 虚拟 Agent

### 原则 B：角色与模式强绑定
- 角色创建时就属于某个空间
- 不再做跨模式切换

### 原则 C：三种角色独立建模
- 现实对话角色
- 剧情角色
- 虚拟 Agent 角色

### 原则 D：三种角色独立记忆
- 不共享长期记忆
- 不共享关系
- 不共享画像

### 原则 E：三种空间独立 UI
- 不能继续混在一个聊天容器里

### 原则 F：不再追求万能角色 schema
- 三类角色结构允许不同
- 只保留底层能力复用，不强求上层字段统一

---

## 十、后续待继续讨论的事项

虽然本轮已经把大的方向定下来了，但还有一批关键问题需要后续单独展开：

### 1. 三类角色各自的正式字段设计
- 现实对话角色需要哪些字段
- 剧情角色需要哪些字段
- 虚拟 Agent 需要哪些字段

### 2. 剧情角色的“世界观层”如何建模
- 是单独一张表
- 还是嵌入角色数据
- 是否支持共享世界观模板

### 3. 三种空间的导航与页面结构
- 首页如何进入三个空间
- 角色列表如何分组
- 会话列表如何设计

### 4. 数据库结构如何调整
- 是否拆表
- 是否增加角色类型字段
- 是否拆出剧情世界观表
- 现有角色数据如何迁移

### 5. 上下文组装器如何拆分
- 三种模式是否使用三套不同的 Context Builder
- 还是同一套框架下三个不同分支

### 6. 记忆提取与关系系统如何分别适配
- 剧情记忆如何存
- 虚拟 Agent 的关系定义是否需要特殊设计
- 现实对话关系是否继续沿用现有体系

---

## 十一、当前复盘总结

本轮讨论最大的收获，不是“又多了一个模式”，而是彻底澄清了以下问题：

1. 三种模式不是风格强弱不同，而是**存在方式不同**
2. 剧情互动不是聊天模式，而是**独立玩法 / 独立空间**
3. 虚拟 Agent 不是折中方案，而是**独立成立的第三种角色类型**
4. 后续系统不应再围绕“一个角色切模式”来设计，而应围绕“不同空间创建不同角色”来设计

这意味着项目后续会从：

> 单一角色系统 + Prompt 分支

逐步转向：

> 三类角色系统并行，底层能力复用，上层结构拆分

这是一次重要的产品结构升级判断。
